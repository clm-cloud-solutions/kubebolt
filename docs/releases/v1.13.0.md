# KubeBolt 1.13.0 — Mode C: agent reads from your existing Prometheus

The 1.13 cycle's flagship: **the agent can now read metrics from a
customer-managed Prometheus** via `/api/v1/query_range`, in addition
to the existing Mode A (kubelet stats directly from the node) and
Mode B (customer's Prom pushes to KubeBolt via `remote_write`). This
closes the deployment patterns where outbound `remote_write` is
either impossible (AWS Managed Prometheus is sink-only) or
change-management-restricted (often the case with GMP-on-GKE and
operators on locked-down Prom configs).

Three production-ready managed-cloud auth providers ship in the
same release:

- **`gcpIam`** — GMP via GKE Workload Identity
- **`awsSigV4`** — AMP via IRSA-bound IAM role
- **`azureWorkloadIdentity`** — Azure Managed Prometheus via AKS Workload Identity

All three were validated step-by-step in-vivo against fresh provider
clusters (sessions 11-A / 11-B / 11-C), first-try after the doc
audit closed.

Backend ships as **KubeBolt 1.13.0**. The agent ships
independently per its own cadence — this release matches it with
**kubebolt-agent 1.1.0**. See the [compat matrix](#compatibility-matrix)
below for the supported pairings.

## ⚠️ Breaking change: `hubble.enabled` default flipped

The agent chart's `hubble.enabled` value default changed from
`true` to `false`. **Cilium users upgrading 1.12.x → 1.13 must
explicitly pass `--set hubble.enabled=true` during the upgrade** or
the agent stops shipping L7 flow data into the Reliability
dashboard.

**Why:** the vast majority of K8s clusters do NOT run Cilium (EKS
with AWS VPC CNI, GKE Standard with kubenet/Calico, AKS with
kubenet, vanilla kubeadm with whatever CNI). Pre-1.13 the agent on
those clusters dialed `hubble-relay.kube-system.svc.cluster.local:80`
forever, emitting one WARN per node per minute indefinitely.
Operators consistently had to override `--set hubble.enabled=false`
to silence the noise. The Reliability dashboard's L7 panels
auto-hide when no Hubble samples reach VictoriaMetrics, so disabling
on non-Cilium clusters loses NO user-visible feature.

**Migration for Cilium clusters:**

```bash
helm upgrade kubebolt-agent oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt-agent \
  --reuse-values \
  --set hubble.enabled=true
```

The chart's NOTES.txt now reflects the off-default — Cilium users
see explicit on-enable guidance, non-Cilium users see no Hubble
noise at install time.

## What ships

### Mode C — agent reads from customer-managed Prom

A new collector pipeline inside the agent (`packages/agent/internal/promread/`):

- **Polls `/api/v1/query_range`** every 30s (configurable) against
  a configured Prometheus-compatible endpoint.
- **Lease-elected single-writer** — the chart renders a separate
  `Deployment` (`replicas=1`, `kubebolt.dev/role=promread`) for
  promread, distinct from the Mode A DaemonSet. Lease name
  `kubebolt-promread` ensures only one pod queries the customer's
  Prom at a time even if the Deployment is scaled past 1.
- **Surgical default matchers** — explicit metric-name lists
  (NOT regex on `__name__`) covering KSM, node load + PSI, node
  disk I/O, node network errors, and target health. Tuned to avoid
  duplicating what Mode A's kubelet collector already provides.
- **Six auth modes** under a single `Provider` abstraction:
  - `none` / `basicAuth` / `bearer` — self-managed Prom
  - `gcpIam` / `awsSigV4` / `azureWorkloadIdentity` — managed cloud

### Backend integration card

- **`/admin/integrations` → "Prometheus (read)" card** flips from
  `Not installed` to `Installed` within ~30-60s of a Mode C agent
  registering. Same install-status pattern as the existing
  Prometheus (push) and Hubble cards.

### Mode A — NodeStress collector (GMP coverage parity)

GMP managed collection does NOT scrape node-exporter, which would
leave the UI's Node Monitor panels (Load average + PSI) empty on
every GKE install. The agent's new NodeStress collector
(`packages/agent/internal/collector/nodestress.go`) reads
`node_load{1,5,15}` and `node_pressure_*_waiting_seconds_total`
directly from `/proc/loadavg` and `/proc/pressure/` on every node,
under `source="kubebolt-agent"`.

The chart auto-defers this collector on Azure (AMW) where
`ama-metrics-node` already scrapes node-exporter, and on bundled
vmagent installs that have node-exporter discovery on. Operators
on AMP with a kube-prometheus-stack `remote_write` source can
opt-in via `--set agent.deferNodeStress=true`. See the
[cross-provider matrix](../integrations/azure-managed-prometheus.md#quirk-4--node_load-and-node_pressure_-double-emission-auto-handled-in-1130).

## What does NOT ship in 1.13.0

- **Mode C against managed Prom without OIDC-style WI** (e.g. Azure
  service-principal-key auth, AWS access-key auth). The agent only
  supports ambient-credential paths for managed clouds — these are
  the more secure defaults and the only ones every managed-K8s
  vendor offers natively. File a request if you need static-credential
  Mode C.
- **Cross-account / cross-subscription / cross-project AMP / AMW / GMP**.
  The agent assumes the workspace is in the same cloud account as
  the cluster. Cross-account adds an IAM trust-policy hop on every
  provider — possible to wire, but out of scope for the doc'd
  recipes.
- **WebSocket-tunneled port-forward** for remote backends — still on
  the 1.14 roadmap. Port-forward works when backend + browser are
  the same machine (UI warns when not).
