# KubeBolt 1.16.0 — Multi-org foundation + operability polish

1.16 continues the **SaaS / Enterprise foundation** started in 1.15 — this
cycle the **Track D** identity seams (organization sign-up, email login,
per-org settings and plans) land as no-op plumbing in single-tenant OSS — while
shipping a round of real OSS operability: an **expanded Add Cluster wizard**,
**always-on insight notifications** for clusters you're not actively viewing, a
**refreshed sign-in**, and an **agent reliability round** (kubebolt-agent
**1.1.2**).

Backend ships as **KubeBolt 1.16.0**; the agent moves to **kubebolt-agent
1.1.2** — a reliability-only bump on the **same metric/label schema**, so it
stays in the **1.1.x generation** and is a drop-in upgrade. See the
[compatibility matrix](../COMPATIBILITY.md) for supported pairings.

> **No breaking changes.** Clean `helm upgrade` from 1.15.x. OSS behavior is
> unchanged: self-service sign-up returns `409 requires_ee` and the sign-up link
> never renders (account creation stays env-var admin seed + in-app Admin → Users
> CRUD). New BoltDB buckets are created automatically. The bundled
> VictoriaMetrics + vmagent stay on `v1.145.0-scratch`.

## Enterprise / SaaS foundation (Track D)

The identity plumbing a multi-tenant SaaS / Enterprise build needs, landed as
no-op seams in single-tenant OSS so the boundary is real before the EE features
exist. Every item below is **inert in OSS** (one implicit `default` org / team).

- **Self-service org sign-up + email login** — `POST /auth/signup` provisions an
  org + default team + admin and logs in, and login accepts email as the global
  identity, with new `/account/plan` + `/account/usage` endpoints. Gated
  `409 requires_ee`; the "Create an account" link only renders when the backend
  reports `signupEnabled` (EE).
- **Org-scoped runtime** — the user's org is stamped into the JWT tenant claim
  and threaded through the user / refresh / team / token / cluster stores; VM
  read queries are tenant-scoped; `/clusters` is filtered by org; settings
  (Copilot / General / Notifications) become org-scoped with a platform-admin
  capability and team-tagged ingest tokens.
- **Anti-spoof ingest** — a `tenant_id` gate on the gRPC ingest path and a
  tenant guard on agent-registry lookups, so an agent can't land samples in the
  wrong org under real token auth.
- **Plans + metering seams** — plan tiers with `ValidatePlan` on org update, a
  no-op api-request metering middleware, per-org cardinality metering hook, an
  agent-ingested-sample meter, and a `UsedSince` period roll-up on the usage
  seam. All no-ops in OSS.
- **Store factory seams** — `AuthStore` / `InsightStore` factory seams and EE
  extension points (routes, nav, backend routes) so EE can back the stores with
  Postgres without forking the server.

## Operability

- **Expanded Add Cluster wizard** — the install flow now surfaces agent
  capabilities that previously required hand-editing the helm command, behind
  progressive disclosure: a **Prometheus integration** choice (kubelet / scrape
  sidecar / **read from your existing Prometheus** — the AMP / GMP / Azure
  Monitor unlock, with per-mode auth: basic / bearer / AWS SigV4 / GCP IAM /
  Azure WI), **mutual TLS** (CA + client-cert Secrets) inline under the Transport
  TLS toggle, ServiceAccount annotations (IRSA / Workload Identity), a
  tolerate-all toggle, a `GOMEMLIMIT` override, and free-form `extraEnv`. Also
  fixes two cases where the generated helm command silently dropped what the UI
  collected (node selector; Hubble relay TLS Secret + SNI).
- **Always-on insight notifications** — notifications now fire for **any**
  registered runtime (active **or** parked / agent-proxy), not just the cluster
  you're currently viewing, while keeping the fresh-episode gate (restored /
  flapping insights don't re-fire). A new `kubebolt_api_runtimes{state}`
  collector exposes the connection pool for dashboards.
- **Delete agent-proxy clusters** from the UI.

## Refreshed sign-in

The login page is rebuilt on a **cinematic split-panel** shell — a dark branding
panel (layered night scene with a breathing aurora, a twinkling starfield, and a
green horizon glow) beside the form, with a lift-and-glow on the sign-in button.
Forced-dark, uses only existing tokens / fonts, honors `prefers-reduced-motion`,
and collapses gracefully to a single-column form below `lg`.

## Kobi fixes

- Never serialize `content: null` toward OpenAI (which rejected the request).
- Surface the **structured chat error body** instead of the raw string.
- Drop the phantom horizontal scrollbar on the chat input in Safari.
- Label fallback sessions with the **provider / model actually used**.
- Resolve the tenant on the public `/copilot/config` route.

## Fixes & security

- **Optional CRDs detected by API discovery**, not RBAC alone — cert-manager
  Certificate, ArgoCD Application, VPA, and the Cilium policy CRDs no longer go
  invisible on clusters where a broad ClusterRole grants the verb but the CRD
  isn't installed.
- **Clear "no cluster selected" message** in place of a generic error.
- **VictoriaMetrics + vmagent stay on `v1.145.0-scratch`** — no third-party
  image change this cycle; the release Trivy gate and Artifact Hub scan stay
  clean with no suppression.

## kubebolt-agent 1.1.2 — reliability

A reliability-only bump on the **same schema** (no UI-visible behavior change):

- **Bounded reconnection** — the gRPC shutdown is time-boxed and the connector
  retry is resilient, and the reconnect backoff is **capped at 3s** after a
  recently-healthy session (it no longer escalates after a brief blip).
- **Homologated memory defaults** across all install paths, with `GOMEMLIMIT`
  derived automatically to drive the Go scavenger.
- **Stable agent id** — derived deterministically so reconnections stop
  accumulating ghost records in the registry.
- **kube-system UID read is retried** before falling back to `"local"`, giving a
  more stable cluster identity on boots behind a slow apiserver.

## Upgrade

```bash
# Backend
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt --version 1.16.0
```

Preserve your overrides by capturing them and re-applying with `-f` (do **not**
use `--reuse-values` — it carries the old chart's bundled-image defaults
forward):

```bash
helm get values kubebolt -n kubebolt-system -o yaml > values.yaml
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  -n kubebolt-system --version 1.16.0 -f values.yaml
```

```bash
# Agent (drop-in within the 1.1.x line)
helm upgrade kubebolt-agent oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt-agent --version 1.1.2
```

New BoltDB buckets auto-create on first boot and OSS behavior is unchanged. The
agent is a same-schema reliability bump — if you're already on 1.1.x there are
no configuration changes required.
