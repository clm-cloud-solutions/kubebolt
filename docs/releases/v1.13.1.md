# KubeBolt 1.13.1 — forward-port of the update-check version stamp fix

Single-fix patch. Forward-ports the same Dockerfile + release.yml
change that landed on the 1.12 line as
[v1.12.2](./v1.12.2.md) so the 1.13.0 container image's
`/api/v1/update-check` endpoint stops surfacing
`{"currentVersion":"dev"}` and starts reporting the actual release
tag.

If you're already on 1.13.0, you'll have noticed the Topbar update
chip never rendered even when 1.13.x+ patches were available — same
root cause as 1.12.1: `apps/api/Dockerfile` was compiled without
the `-ldflags="-X main.version=<tag>"` stamp the standalone binaries
get, so `main.version` (in `apps/api/cmd/server/main.go`) kept its
default `"dev"`, and the update-check package short-circuits on a
`"dev"` binary.

1.13.1 closes that gap so the chip works on 1.13.x installs going
forward.

---

## What ships

### Backend

- **`apps/api/Dockerfile`** — adds `ARG VERSION=dev` and rewires the
  `go build` to `-ldflags="-s -w -X main.version=${VERSION}"`.
  Identical to the v1.12.2 change.
- **`.github/workflows/release.yml` `build-api` job** — passes
  `build-args: VERSION=${{ steps.meta.outputs.version }}` to
  `docker/build-push-action`. Identical to the v1.12.2 change.

No agent changes — the agent (`kubebolt-agent`) stays at `1.1.0`
shipped in the 1.13.0 release. The bug was always api-image-only.

The single-container image (`Dockerfile.single`) was unaffected on
both lines — it consumes the pre-built binary from `build-binaries`
which always had the `-ldflags` stamp.

### Release workflow — GitHub Release page is now human-friendly

The Topbar update chip points at
`https://github.com/clm-cloud-solutions/kubebolt/releases/tag/v<version>`,
which previously rendered only install instructions plus an
auto-generated commit-log changelog — too technical for someone
landing there because they clicked a "what's new?" chip.

The `release` job now prepends `docs/releases/v<tag>.md` to the
Release body when the file exists. That's the narrative
walkthrough humans actually want to read — what shipped, why,
upgrade commands, breaking changes called out. Install
instructions + commit-log changelog still appear below, so power
users have everything in one place.

This kicks in starting with **this release** (1.13.1). The 1.13.0
and 1.12.2 GitHub Release pages keep their original install-only
bodies; if you want the narrative on those, the markdown files
live at the same URLs (`docs/releases/v1.13.0.md`,
`docs/releases/v1.12.2.md`) and can be opened directly.

## Verification

After upgrading the backend image to `1.13.1`:

```bash
curl -sH "Authorization: Bearer <admin-jwt>" \
  https://<your-host>/api/v1/update-check | jq
# Expected:
# {
#   "enabled": true,
#   "currentVersion": "1.13.1",
#   "latestVersion": "1.13.1",        ← or a higher patch when one ships
#   "isUpdateAvailable": false,
#   ...
# }
```

The Topbar update-check chip will surface as soon as a higher patch
is available on GitHub Releases.

## Upgrade

### From 1.13.0

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  --version 1.13.1 \
  --reuse-values
```

Zero-config impact — same chart values, same env vars, same RBAC.
No agent re-deploy needed; `kubebolt-agent 1.1.0` is the pair for
both 1.13.0 and 1.13.1.

### From 1.12.x

Read [v1.13.0](./v1.13.0.md) first — it has Mode C, the three
managed-cloud auth providers, and **a BREAKING change** for Cilium
clusters (the `hubble.enabled` default flipped). 1.13.1 inherits all
of those; upgrading directly from 1.12.x to 1.13.1 is supported but
the BREAKING change still applies.

## What does NOT ship in 1.13.1

## Why two patch releases for the same bug

The 1.12.2 patch shipped first because that's the line every
self-hosted install was already running when the bug was
discovered. Forward-porting to 1.13.1 immediately after lets
operators on 1.13.0 either:

- Patch in place via `--version 1.13.1`
- Or skip to 1.13.1 if they hadn't yet upgraded from 1.12.x

Both 1.12.2 and 1.13.1 carry the same fix verbatim — there is no
risk of behavioral divergence between the two patches.
