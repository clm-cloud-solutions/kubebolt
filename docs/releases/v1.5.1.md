# KubeBolt 1.5.1 — Auto-compact that actually compacts

Patch release. If you shipped 1.5.0 and saw the Copilot's context
counter creep past the 80% threshold without anything happening — or
worse, saw the Copilot answer "the output was truncated" mid-session —
1.5.1 is the fix. Three linked bugs in the auto-compact flow now
addressed end to end, with regression tests to keep them fixed.

Upgrade and move on.

## What's fixed

### The trigger actually fires on round 0 of follow-up questions

In 1.5.0 the UI showed `Session: 28k / 24k` but the next question
didn't compact. Reason: the backend's own estimate of context size
ignored the system prompt and tool definitions (~15–20k tokens) and
underestimated JSON-dense tool results. It saw ~20k, didn't cross the
trigger, skipped compact. You had to send two or three filler messages
to nudge it over.

1.5.1 carries the provider-reported usage of the previous round as a
hint from the frontend to the backend, so the check starts from the
accurate number the UI already shows. A server-side estimate of the
system + tools overhead also closes the gap when no hint is available
(first request of a fresh session).

### Compacts now free the bulk, not just the skeleton

Before: `1 turn folded · 25k → 25k`. Compact ran, folded an old user
message plus its short assistant reply, and left the heavy tool
results — which live in the most recent preserved turns — completely
untouched. The context stayed heavy and the trigger fired again
immediately.

After: compact still folds older turns into a summary *and* replaces
bulky tool_results in the preserved tail with short stubs. The LLM
keeps your questions, the assistant's synthesis, and the record of
which tools were called — it just loses the raw data it can re-fetch
in one tool call. The compact already invalidates the provider prompt
cache via the new summary prefix, so losing cached tail bytes adds no
extra cost.

Typical new pattern in the banner: `AUTO-COMPACTED · 2 turns folded + 3
tool results stubbed · 40k → 9k (−78%)`.

### The Copilot no longer says "the output was truncated"

In 1.5.0 a compact firing mid-request between rounds could stub the
tool_results the LLM had *just received* and was about to read.
Result: the LLM saw placeholders and apologised for the truncated
data. 1.5.1 computes an "active turn" boundary — the last user-text
message — and never stubs anything from that index onward. Two
regression tests lock the behaviour in:

- `TestCompact_ProtectsActiveTurnToolResults` — verifies the active
  turn's raw bytes survive byte-for-byte.
- `TestCompact_StubsNonActiveTailToolResults` — verifies turns already
  synthesized *do* get stubbed.

## Nice to have

- Short conversations heavy on a single tool call (think `get_pod_logs`
  pulling 50 KB) now benefit from compaction too. When there aren't
  enough turns to fold, Compact stubs earlier tool_results in place
  and reports how many.
- The Copilot panel banner tells you what actually happened: "N turns
  folded", "N tool results stubbed", or both when both apply.
- A new `copilot auto-compact noop` log line explains the cases where
  the trigger fires but neither folding nor stubbing is possible — no
  more mystery triggers in the admin view.
- `ToolResultsStubbed` travels in the SSE `compact` event, the
  `/copilot/compact` response, and the persisted session record.

## Upgrade

### Helm

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt --version 1.5.1
```

### Docker Compose

```bash
cd deploy && docker compose pull && docker compose up -d
```

### Homebrew / krew

```bash
brew upgrade clm-cloud-solutions/tap/kubebolt
kubectl krew upgrade kubebolt
```

## Configuration

No new environment variables. All the existing `KUBEBOLT_AI_AUTO_COMPACT*`
knobs from 1.5.0 still apply with the same defaults.

## Known issues (unchanged)

- `/cluster/overview` still returns zero for `health.insights.*`
  counters. Scoped for a later release.
- Tool-result JSON truncation is still byte-aligned; structure-aware
  truncation remains on the roadmap.

## Thanks

To every tester who caught "trigger without applied" and "output
truncated" in the logs and sent screenshots. This release is tighter
because of the noise you made.
