# KubeBolt 1.12.1 — UI update-check chip + air-gapped opt-out

Single-feature patch. A discreet **"Update vX.Y.Z available"** chip now
appears in the Topbar when a newer stable KubeBolt release is
published on GitHub, with a link to the release notes. Lands in
1.12.x specifically so users already running 1.12 see the prompt
when 1.13 drops — shipping the same feature in 1.13 itself would
only benefit 1.14+ announcements, where the 1.13 launch gets no
visibility uplift.

Air-gapped operators disable the whole thing with one env var or one
admin toggle — the backend never reaches `api.github.com` when off.

---

## What ships

### UI

- **Topbar chip** mounted alongside the existing
  `PortForwardIndicator`. Renders only when a higher stable semver
  is available on GitHub — no chip when up to date, no chip on dev
  builds, no chip when disabled. Click the version opens the release
  notes (`https://github.com/clm-cloud-solutions/kubebolt/releases/tag/v<version>`)
  in a new tab; click the X dismisses for that specific version
  (a future v1.13.0 re-surfaces the chip even after dismissing
  v1.12.1).
- **Settings → General → "Update check" card** with a single toggle.
  Same runtime-config pattern as the other 1.12 General-tab knobs
  (display name, refresh interval, prod namespace pattern):
  BoltDB-persisted override that wins over the env baseline.

### Backend

- **`GET /api/v1/update-check`** — auth required (any role). Returns
  `{enabled, currentVersion, latestVersion, isUpdateAvailable, releaseUrl, releaseName, publishedAt}`.
  Cache TTL is 6h, so even at heavy frontend polling, real GitHub
  traffic stays at ~4 requests/day per backend — well under the 60/h
  unauth rate limit for shared egress NAT.
- **`internal/updatecheck/` package** — defensive: filters
  `prerelease` and `draft` releases out, picks the highest semver
  (not the first chronologically), swallows GitHub errors so the UI
  just doesn't render the chip instead of throwing, and
  short-circuits entirely when the binary's `main.version == "dev"`
  (local development never reaches GitHub).

### Air-gapped

- **`KUBEBOLT_UPDATE_CHECK_ENABLED=false`** (env var) OR admin
  toggle off OR Helm `extraEnv`. Any of the three stops all outbound
  traffic to `api.github.com`. Documented in all four env templates
  (`.env.example`, `deploy/.env.example`, `deploy/docker-compose.yml`,
  `deploy/helm/kubebolt/values.yaml` `extraEnv` example).

PR #65 (feature), PR #66 (develop → main).

## Why a separate patch and not a 1.13 warm-up

For "tell users about 1.13" to actually work, the check has to exist
in the version users are running BEFORE 1.13 drops — that's 1.12.x.
Shipping the check in 1.13 itself only helps notify 1.13-and-later
users about 1.14+ releases; the 1.13 launch itself gets no
visibility uplift from the feature.

Sized as a patch (not a minor) because the scope is one feature,
contained to a known surface, with no breaking changes and no
schema migrations. 1.13 stays single-focus on Mode C.

---

## Upgrade

Single helm upgrade — no values changes required, no agent restart
needed, no breaking shape:

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  --version 1.12.1 \
  --namespace kubebolt-system \
  --reuse-values
```

The agent chart (`kubebolt-agent`) stays at **1.0.0** — no DaemonSet
code touched in this patch.

Existing installs see the chip appear in the Topbar within ~6h of
the GitHub release going live (or immediately on first request after
backend boot). The admin General tab gains the new "Update check"
card.

## What does NOT ship in 1.12.1

- **Mode C** — agent reads customer's Prometheus (AMP / Azure
  Managed / GMP). Still the 1.13 flagship, full design + sprint
  plan in `internal/roadmap-1.13.md` (private).
- **Auto-upgrade** — the chip only informs. Upgrades remain manual
  via `helm upgrade`.
- **Email / Slack notifications on new release** — out of scope.
  The chip is the only notification surface for now.
- **Changelog inline in the UI** — the link to GitHub release notes
  is the deliverable; no server-side markdown parsing.

## Validated against

- Local dev with version stamped to `v1.11.0` (vs. `v1.12.0` latest
  on GitHub) — chip appeared as expected, click opened release
  notes, X dismissed with localStorage persistence, toggle off
  pushed `{enabled: false}` to the endpoint and removed the chip.
- Unit tests pin the prerelease/draft filter, the highest-semver
  pick (not first-in-list), git-describe semver shapes
  (`v1.12.0-2-gabc1234`, `v1.12.0-dirty`), and the dev-build
  short-circuit.

## Out of scope (tracked separately)

- Mode C (1.13 flagship — sized at 3.5-4 weeks).
- mTLS / Cortex tenant headers / OIDC refreshable tokens — fast-
  follow if customer demand emerges.
