# KubeBolt 1.12.0 — Kobi gets metrics, admin config moves to UI, Pod Debug + Ingest Activity

No single flagship dominates this release — the cohesion is
**operator/admin self-service**. Across eight merged PRs and four
workstreams, KubeBolt closes four gaps that all push toward the same
direction: less env-var configuration, more in-UI control, deeper
introspection without leaving the dashboard. Copilot becomes a
metrics first-class citizen ("ask Kobi why CPU spiked yesterday"),
admin runtime configuration moves out of restart-required env vars,
on-call gets ephemeral debug containers for distroless workloads, and
the ingest pipeline gains its own live observability panel.

Mode C (agent reads customer's Prometheus, for AMP / Azure Managed
Prom shops) was originally framed as the 1.12 flagship but slipped to
**1.13** during the cycle — the four workstreams below took
precedence after in-vivo validation surfaced higher-impact gaps. Mode
C remains the 1.13 anchor.

---

## Headline workstreams

### 1. Copilot (Kobi) becomes metrics-aware

Four PRs (#55 → #58) chain into one capability: **"ask Kobi why CPU
spiked yesterday at 3pm"** now works end-to-end with an inline chart
in the analysis turn.

What landed:

- **`get_workload_metrics` tool** — Kobi can pull CPU / memory /
  network / disk PromQL trends scoped to a workload (Deployment,
  StatefulSet, DaemonSet, Job, **Node**) and a time range. Uses the
  same PromQL builders the Capacity dashboards use, with shared
  `label_replace` chain to collapse pod-keyed Hubble metrics into
  workload-keyed series. Roll-up math is correct for all five kinds
  including the StatefulSet edge case (no hash suffix to strip).
- **`KobiMetricChartCard` inline rendering** — when the tool returns
  a metrics payload, the chat surface renders the chart **inside the
  analysis turn** (not as a follow-up message). One small fix
  (`05db24c`) re-anchors the card to the analysis turn instead of
  the raw tool-call turn, so the chart shows up where the
  explanation lives.
- **Node kind support** — covers the "is the node hot?" path that
  every cluster investigation eventually reaches. Matches the
  Monitor tab's PromQL exactly so Kobi's numbers agree with what the
  operator sees in the UI.
- **SessionRecord persists the resolved model** — small fix (#55):
  the per-session model recorded against each turn now reflects the
  default-model resolution at write time, so sessions started before
  a tenant configured an explicit model still show the actual model
  used (instead of a blank field).

PRs #55, #56, #57, #58. Unit tests pin the workload roll-up matrix.
Validated in-vivo against a kind-kubebolt-dev cluster with real CPU
spike patterns.

### 2. Admin runtime configuration moves out of env vars

Two PRs (#59 + #60) deliver spec #09 V1 + V2 — operators stop
needing to restart the API to change runtime knobs.

V1 (PR #59) shipped:

- **First-login wizard** — three-step onboarding when the admin
  account logs in to a fresh install for the first time. Walks
  through display name / refresh interval (General), JWT lifetime /
  refresh-token policy (Auth), notification channel toggles
  (Notifications). Replaces the previous "go read four env vars
  before you can use the product" experience.
- **`/booted-with` diagnostic** — one new GET endpoint surfaces
  exactly which env vars the API actually read at boot (vs which it
  fell back to defaults on). Crucial for "why isn't my override
  taking effect?" debugging.
- **Five tabs in Settings** — General / Auth / Notifications /
  Copilot / Ingest. Every knob that was previously a restart-
  required env var now has a UI control with persist-to-BoltDB +
  hot-reload through the runtime config system.

V2 (PR #60) closed the gaps:

- **"Agents & Ingest" tab refactor** — the Ingest tab is now a
  grid of five sub-cards (Channel auth, Rate limits, Tenants,
  Per-tenant limits, Ingest tokens) with a single unified Save bar.
- **BootedWithModal** categorizes env vars by **BOOT-TIME** (must
  set before first start) vs **Runtime** (can be changed in UI
  later), so admins know which side of the line they're working
  with.
- **Wizard step 3 auto-issues an ingest token** when channel auth
  is set to `enforced` — closes the gap where a fresh install
  immediately went into a state where vmagent couldn't authenticate
  to the receiver.
- **`prodNamespacePattern`** added to General settings — a regex
  (with live preview) that flags production namespaces in
  notifications, the topology map, and the resource list views.
- **Env templates reorganized** — `.env.example` × 4 (root, deploy/,
  apps/api/, packages/agent/) grouped into BOOT-TIME / Runtime
  tiers matching the bucket categorization. A fresh install can now
  leave the entire RUNTIME tier empty and configure the system
  end-to-end through the wizard + Settings UI.

Per-tenant limits, regex preview, hot-reload — all the surface area
needed to run KubeBolt without ever touching env vars after the
initial helm install.

PRs #59 and #60.

### 3. Pod Debug — ephemeral container for distroless workloads

PR #61 closes the most common "I can't get a shell in this pod"
complaint. Distroless containers have no shell binary, so the
existing Terminal tab fails immediately. KubeBolt now ships an
ephemeral-container debug flow:

- **DebugPodModal** — operator picks an image (default `busybox`,
  alternatives `nicolaka/netshoot`, `alpine`), confirms the target
  pod + namespace, and KubeBolt attaches an ephemeral container via
  `kubectl debug` semantics.
- **Terminal tab auto-selects the debug container** — when an
  ephemeral debug container exists on a pod, the Terminal tab's
  container selector defaults to it instead of forcing the operator
  to pick from the dropdown. Markers in the dropdown distinguish
  debug containers from app + init containers.
- **URL race fix** — the post-modal navigation to
  `?tab=terminal&container=debug-XYZ` previously raced the page
  load and sometimes resolved before the new container was
  registered in the pod spec. Fixed via a small retry shim on the
  container-list fetch.
- **Backend handler** — `POST /resources/pods/:ns/:name/debug`
  with `kubectl debug` semantics under the hood. Editor role
  required.

PR #61. Item 4 of the 1.12 roadmap.

### 4. Ingest Activity panel — closes Workstream B

PR #62 ships the operator-facing answer to *"what is my ingest doing
right now?"* — closes Workstream B that started in 1.11 with the
Prometheus integration card.

At `/admin/ingest-activity`:

- **Per-tenant card** with samples/sec headline + **active series
  gauge vs cap** (color-coded warn ≥70%, error ≥90%) + tenant UUID
  + Default-tenant pill.
- **1h sparkline** (RangeSelector lets you switch 5m / 15m / 1h /
  6h / 24h / 7d uniformly) split into **two series**: agent gRPC
  ingest vs Prometheus remote_write. Operator can see at a glance
  which channel is carrying the load.
- **Status chips** for stream lifecycle (connected / disconnected /
  reserved auth_rejected) and remote_write request outcomes
  (accepted / auth / rate-limit / cardinality-capped / malformed /
  other rejected). Chip windows scale with the RangeSelector.
- **Live heartbeat list** of currently-connected agents — reads
  the in-memory `AgentRegistry` directly via a new `/admin/agents`
  endpoint, so freshness is sub-second instead of 30s VM scrape
  lag.

Architecturally — and this is the operationally interesting part —
the panel is **self-contained**. No external scraper required.
The backend self-writes its own Prometheus counters to VM every 30s
via `/api/v1/import/prometheus`, and a new admin PromQL endpoint
bypasses the cluster-scoping middleware so tenant-scoped metrics
(which don't carry `cluster_id`) are queryable. Works identically
in OSS single-cluster installs and SaaS multi-cluster topologies
where backend and customer cluster live on different networks.

PR #62. Item 5b of the 1.12 roadmap.

## Upgrade

Single helm upgrade — no values changes, no agent restart needed,
no breaking shape:

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  --version 1.12.0 \
  --namespace kubebolt-system \
  --reuse-values
```

The agent chart (`kubebolt-agent`) stays at **1.0.0** — the DaemonSet
agent shipped no functional code changes in this cycle (only an
`.env.example` reorganization). Operators running a separate agent
chart do **not** need to upgrade it.

First boot on a fresh install will trigger the new wizard for the
admin account. Existing installs see the new sidebar entries
(**Ingest Activity** under Admin, **Settings → Agents & Ingest**)
and the BootedWithModal accessible from the Settings header.

Custom runtime knobs you've previously configured via env vars
continue to work — the UI controls layer on top, with the
BootedWithModal showing which value won (env vs UI). Hot-reload
applies through the runtime config system on save.

## What does NOT ship in 1.12

- **Mode C** — agent reads customer's Prometheus, for AMP / Azure
  Managed Prom shops. Diferred to 1.13 as the next flagship. The
  decision happened during 1.12's cycle when in-vivo validation of
  the first three workstreams surfaced enough refinement scope that
  Mode C couldn't land cleanly.
- **`auth_rejected` gRPC stream counter** — the label is reserved
  in the metric but not wired into both interceptors backend-side.
  WARN log on rejection covers diagnostics in the meantime. Chip
  only renders when count > 0. Full wiring tracked for V2 in the
  spec.

## Validated against

- `kind-kubebolt-dev` (local kind cluster, 3-node, mixed workloads)
  — every workstream validated end-to-end in vivo before merge.
- The Copilot chain (#55 → #58) additionally validated against
  `yagan-eks-prod-v2` (EKS 1.34, kube-prometheus-stack) — the
  Workmate-client production cluster from 1.11.1.
- The Ingest Activity panel's self-write architecture was the
  output of a real in-vivo failure (cross-network scrape from
  vmagent in a customer cluster to a self-hosted backend timed
  out) — the pivot to API-writes-to-VM came directly from that
  failure mode.

## Out of scope (tracked separately)

- Mode C (Workstream C of 1.13).
- `auth_rejected` chip wiring backend-side (V2 of spec #09 follow-
  up).
- Agent chart bump — no DaemonSet code changes in this cycle, so
  `kubebolt-agent` stays at 1.0.0.
