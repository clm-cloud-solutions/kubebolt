# KubeBolt 1.11.1 — Real-world cluster polish

A patch release that closes five gaps surfaced during a real-cluster
install validation campaign on `yagan-eks-prod-v2` (Workmate client,
EKS 1.34 with kube-prometheus-stack + AWS VPC CNI + 6 vmagent
DaemonSet pods). Every fix here was triggered by something the
operator saw with their own eyes — empty endpoint cells, missing
sidebar counters, vmagent error storms — and traced back to a
specific gap between what KubeBolt assumes about the world and what
the world actually ships with.

No new features, no breaking changes. Pure quality-of-life for the
clusters that don't look exactly like the development testbed.

---

## What's fixed

### 1. UI: Services no longer all render as "down" on stock kube-prometheus-stack

The Service list and detail views read endpoint counts from
kube-state-metrics. KubeBolt's hook queried only the modern
`kube_endpointslice_*` collector. **kube-prometheus-stack** — the
most common Prometheus install in the wild — ships its bundled KSM
with the legacy `kube_endpoint_*` collector enabled instead. Result:
zero series, empty hook, every Service rendered as `down` (red)
even when healthy.

The hook now queries both families in parallel and merges, preferring
modern when both are present. Works out-of-the-box on:

- standalone `prometheus-community/kube-state-metrics` clusters
- stock `kube-prometheus-stack` clusters (the yagan case)
- clusters that enabled both collectors during a migration

PR #49 · 7 vitest cases pinning the modern-only / legacy-only / mixed
/ empty matrix.

### 2. UI + API: Gateways / HTTPRoutes / Endpoints sidebar counts + Endpoints list + Endpoints YAML

Three coordinated holes in the backend type-keyed dispatchers all
hit the same resource family:

- **`ClusterOverview` struct** had no fields for these three counts,
  so the sidebar's count chip stayed hidden even when the resources
  existed (`Gateways=1`, `HTTPRoutes=10`, `Endpoints=56` in the
  yagan validation cluster).
- **`GetResources` LIST switch** had cases for `gateways` and
  `httproutes` but **no `case "endpoints"`** — falling to default
  returned `Total: 0` even with 56 EndpointSlices in the cluster.
  The `/endpoints` page rendered "No endpoints in this cluster"
  while sidebar said 56.
- **`resourceTypeToGVR` map** — the central GVR table used by
  YAML view, Delete, Create, PatchMetadata, and ApplyYAML — had no
  `"endpoints"` entry. Every YAML tab click on an EndpointSlice
  errored with "unsupported resource type: endpoints" from five
  different handlers.

Fixed in one coordinated PR: struct field, list helper, mapper, and
the central GVR entry that unblocks five dispatchers at once.

PR #50.

### 3. Copilot: `get_pod_logs` no longer wastes a round on multi-container pods

Calling `get_pod_logs` on a multi-container pod (gitlab-webservice
has 5 containers; every istio-injected pod has 2+) without
specifying `container` previously failed with the apiserver's raw
error and the LLM had to parse English to retry — sometimes
picking wrong and finding out only when the second call's contents
arrived.

The tool now follows a **list-then-pick** pattern:

- Single-container pod: auto-resolve transparently.
- Multi-container + `container` omitted: return **only the
  container list** (no logs) and nudge the LLM to pick using its
  world-knowledge of container-naming conventions (skip
  `certificates` / `configure` / `dependencies` init helpers;
  prefer the app container by name; pick `istio-proxy` / `vault-
  agent` only when the user's question is about that concern).
- Multi-container + `container` supplied: validate against the
  pod's actual containers list. Mismatch surfaces a structured
  error with `availableContainers` so the LLM corrects in one
  round (no apiserver 404).

Net savings: **~50% on common multi-container investigations**
(skips the "wrong container's logs" round that auto-pick produced).

PR #51.

### 4. Auth: custom ingest limits now actually apply to permissive traffic

The most operational fix in this release. Operators who set custom
Write rate / Burst / Max active series on the "Default tenant" in
the UI saw 429 storms in vmagent's logs anyway, with the rate
limiter reporting `tenant_id="anonymous"` accepting and rejecting
requests in droves. The operator's custom limits were silently dead
code because:

- The UI's "Default tenant" is a real BoltDB tenant record
  (UUID-named).
- The rate limiter's per-request tenant key comes from the request
  Bearer token, or stamps `"anonymous"` when there's no valid
  token.
- In a default install (`agentIngest.authMode=permissive` + no
  ingest token provisioned), vmagent's writes carry no Bearer
  header → backend resolved them as `"anonymous"` → rate limiter
  looked up overrides for `"anonymous"`, found none, applied fleet
  defaults (10k/s + 100k burst).

The receiver now auto-resolves unauthenticated traffic to the
install's default tenant when `TenantsStore` is wired (the common
OSS install). Custom overrides on that tenant **apply automatically
to disabled / permissive flows** — no operator action beyond setting
the limits in the UI.

The decision rule encoded by this change, preserved verbatim in
code comments:

> KubeBolt auth enabled → TenantsStore exists → tenant_id MUST be
> present on every write (either supplied or auto-assigned default).

The `tenant_id="anonymous"` label disappears from internal metrics
for unauthenticated traffic in healthy installs. The edge case of
`KUBEBOLT_AUTH_ENABLED=false` (no TenantsStore) preserves the
legacy bypass — same behavior as before.

PR #52 · 9 HTTP tests + 6 gRPC tests pinning the full matrix
(disabled/permissive/enforced × with/without TenantsStore ×
valid/invalid/missing Bearer).

**Practical effect on your install**: the moment you bump the
backend image to 1.11.1, any custom limits you've configured under
Administration → Ingest Limits start being honored for vmagent's
writes. No restart of the agent needed; the rate limiter pulls the
new resolution on the next request.

### 5. Docs: chart NOTES warn about Hubble retry loop on non-Cilium clusters

The agent's `values.yaml` and `NOTES.txt` previously claimed that
Hubble flow collection "silently no-ops" when Cilium isn't
installed. **Not true**: the agent dials `hubble-relay.kube-
system.svc.cluster.local:80` forever, DNS returns NXDOMAIN, and
the agent emits a WARN per node per ~1m indefinitely. Operators
saw the WARN spam during the yagan validation and reasonably
assumed something was misconfigured.

Chart comments + NOTES.txt now flag the behavior explicitly,
recommend `hubble.enabled=false` for AWS VPC CNI / kubenet /
non-Cilium GKE installs, and explain that the Reliability dashboard
auto-hides itself on those clusters anyway — disabling Hubble loses
zero user-visible feature, only the noise.

PR #48.

## Upgrade

Single helm upgrade — no values changes, no agent restart, no
breaking shape:

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  --version 1.11.1 \
  --namespace kubebolt-system \
  --reuse-values
```

The agent chart (`kubebolt-agent`) stays at `1.0.0` — no version bump
needed, this release only changes backend code + UI code + chart
documentation.

The custom-limits fix (#52) and the endpoints-collector fix (#49)
both take effect at the next request / next scrape cycle after the
new pod is Ready — no operator action beyond the upgrade itself.

## Validated against

`yagan-eks-prod-v2` (EKS 1.34, 6 nodes, kube-prometheus-stack +
AWS VPC CNI, 84 pods across 17 namespaces) — the same cluster that
surfaced every fix in this release.

## Out of scope (tracked separately)

The agent itself still emits a WARN per Hubble retry on non-Cilium
clusters. Demoting that to DEBUG after the first failure is an
agent-runtime change that didn't make this cut. Tracking as a
follow-up.
