# KubeBolt 1.17.1 — split-origin (remote UI) hardening + Gateway API deploy path

1.17.1 is a patch on 1.17.0 that hardens deployments where the **UI is served
from a different origin than the API** (a static/edge host talking cross-origin
to a remote API) and adds a **Gateway API** deploy path to replace the retiring
`ingress-nginx`. Drop-in over 1.17.0 — **no schema change**. Same-origin installs
(the default) are unaffected.

---

## Gateway API (Envoy Gateway)

`ingress-nginx` is being retired. New chart templates, gated on
`gatewayAPI.enabled` (mutually exclusive with `ingress`):

- **Gateway** (GatewayClass `eg`) with two HTTPS listeners — the API (HTTP/REST +
  WebSocket) and the agent gRPC AgentChannel — TLS-terminated, certs issued by
  cert-manager.
- **HTTPRoute** (API) + **GRPCRoute** (agent-ingest); the agent-ingest Service
  carries `appProtocol: kubernetes.io/h2c` so the gateway speaks gRPC upstream.
- A **`ClientTrafficPolicy`** pins the **API** listener ALPN to `http/1.1`
  (Envoy's default HTTP/2 breaks the WebSocket upgrade and resets streams under
  bursts). It is scoped to the API listener **only** — the agent gRPC listener
  keeps `h2` (else agents fail their handshake with *"missing selected ALPN
  property"*). Toggle with `gatewayAPI.forceHTTP1` (default true).

Front the chart with a Gateway API controller (we use Envoy Gateway) and set
`gatewayAPI.enabled=true`.

## Cross-origin (remote UI) fixes

- **CORS** now allows the custom `X-KubeBolt-*` request headers
  (`X-KubeBolt-Cluster` for cluster selection + the Kobi context headers), so the
  cluster-scoped and Kobi requests work when the UI is on a different origin than
  the API. `VITE_API_URL` / `KUBEBOLT_CORS_ORIGINS` remain the knobs (from 1.17.0).
- **Port-forward** links build against the API origin instead of a relative
  `/pf/{id}/` (which, cross-origin, resolved to the UI host instead of the API's
  reverse-proxy).

## Security

- The `/pf/{id}/` reverse-proxy is now **authenticated** when auth is enabled: it
  was reachable with only the unguessable id. A top-level browser navigation can't
  carry a Bearer header, so access is authorized by a path-scoped, HMAC-signed
  `kb_pf` cookie (HttpOnly, Secure, SameSite=Lax, 8h) issued at port-forward
  create. The auth-disabled path (the default single-binary dev case) is unchanged.

## Upgrading

Drop-in over 1.17.0 — no schema change. The CORS / port-forward changes are no-ops
when the UI and API share an origin. For a Gateway API deploy, set
`gatewayAPI.enabled=true` and install a Gateway API controller.
