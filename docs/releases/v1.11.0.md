# KubeBolt 1.11.0 — NetworkPolicy first-class + Prometheus integration surface

This release closes five items from the post-1.10 roadmap. The
headline is **NetworkPolicy as a first-class resource** — a gap that
showed up in the first five minutes of every regulated-industry
cluster validation and the largest "kubebolt doesn't show me what
kubectl shows me" delta we had left. Alongside it ship: Prometheus
promoted to a first-class integration in the admin UI, per-cluster
scoping for ingest tokens (so multi-cluster Prom setups don't bleed
samples across cluster_id boundaries), a Reliability tab that
honestly splits the L4 vs L7 story when Hubble L7 isn't available,
and a docs path for GKE Dataplane V2 managed Hubble cert
distribution.

Mode C (agent reads customer's Prometheus, for AMP / Azure Managed
Prom shops) is captured for **1.12** as the next minor's flagship —
out of scope here.

---

## Headline features

### 1. NetworkPolicy as a first-class resource (Workstream A — flagship)

NetworkPolicy joins the 23 Kubernetes types KubeBolt renders with
full list + detail + sidebar + insights coverage. This is the type
regulated-industry and multi-tenant clusters notice missing in the
first five minutes; the gap was identified empirically during the
1.10 cluster-validation campaign and committed to the 1.11 roadmap
as Item 1.

What landed:

- **Typed informer + lister** for `networking.k8s.io/v1.NetworkPolicy`
  with the same per-namespace SA support every other resource type
  gets.
- **List view** with custom columns: pod selector preview (catch-all
  highlighted in amber), policy types, ingress + egress rule counts.
- **Detail view** with a dedicated **Matched Pods** tab that answers
  the operator's first question on any policy — *"is this thing
  actually gating anything?"* — by running the policy's `podSelector`
  against the namespace's pods server-side and returning the match
  set + count. Three empty-state shapes distinguish "RBAC denied"
  vs "policy is catch-all and namespace is empty" vs "selector is
  specific and matches zero pods" so the operator never has to guess
  why a list is empty.
- **Two insights rules** join the engine — `policy-no-match`
  (warning: a podSelector that matches zero pods, almost always a
  selector-key typo) and `policy-orphan` (info: a namespace with
  running pods and zero NetworkPolicies, with a system-namespace
  skip list so platform namespaces don't generate noise).
- **New-resource modal** offers three curated starter templates
  ordered by typical first-touch frequency: *Default deny all*,
  *Allow same namespace*, *Allow from specific app* (the working
  illustration of `podSelector` + `ingress.from.podSelector` that
  most newcomers get wrong on the first try).
- **Cmd+K global search**, **notifications deep-links**, and Helm
  chart **RBAC** all updated symmetrically — full surface coverage,
  no half-finished corners.

Insights rule count goes 13 → 15. Resource-type RBAC probe count
goes 22 → 24 (`networkpolicies` plus the `replicasets` informer
that was already present but not counted).

### 2. Prometheus as a first-class integration (Item 5a)

The remote_write receiver landed in 1.10 (Phase 3). 1.11 makes the
relationship **visible**. Prometheus now appears in
`/admin/integrations` alongside the agent with status + version +
last-heartbeat, surfacing the answer to *"is my remote_write actually
connected and streaming?"* — previously the operator had to inspect
`kubebolt_prom_write_*` counters in `/metrics` or out-of-band logs.

Detection blends two signals:

- **Primary**: in-cluster workload presence — pods labeled
  `app.kubernetes.io/name=prometheus`. The vast majority of installs
  (kube-prometheus-stack default) put Prom in the same cluster they
  monitor.
- **Secondary**: ingest-token heartbeat — for the federation /
  managed-Prom case where Prometheus pushes from outside the
  cluster KubeBolt monitors.

The card surfaces install hints (annotation-driven discovery
recipes) and a copy-pasteable `remote_write` snippet pre-filled with
the active cluster's `cluster_id`, `cluster_name`, and `tenant_id`
labels so the snippet works without manual edits.

### 3. Per-cluster ingest token scoping (Item 5a.1)

Multi-cluster Prom setups need each cluster's samples to carry that
cluster's `cluster_id` label so the dashboards don't collapse them
into one. The IssueToken admin flow now lets you pick *which* cluster
a token is scoped to:

- The token's `IngestToken.ClusterID` is persisted in BoltDB and the
  backend stamps that label on every sample written through it
  (overriding whatever the Prom sender labels itself).
- The admin UI ClusterScopeDropdown shows scoped clusters first,
  then unscoped clusters as a hint ("visit this cluster first to
  scope its token here"), then an "any cluster" option as the
  legacy unscoped path.
- The Prometheus card on the Integration detail page reads each
  cluster's `cluster_id` from a **per-cluster sample-presence
  probe** (`count(up{cluster_id="X"})` against VM) instead of just
  presence-or-absence of a token. A legacy token from before this
  feature that matches every cluster no longer falsely shows
  "Streaming" on clusters that aren't actually sending — the probe
  honors the per-cluster signal.
- The active cluster's `cluster_id` (kube-system namespace UID) is
  now persisted in BoltDB on first connect and exposed via
  `ClusterInfo.clusterId`, so direct-kubeconfig contexts (not just
  agent-proxy contexts) can pre-fill the snippet correctly.

### 4. Reliability tab L4/L7 honest split (Item 3)

The Reliability tab previously hid itself completely when Hubble L7
samples weren't streaming. That was a sensible default while the
feature was new, but it had a downside: customers running Cilium L4
*without* Hubble L7 (Cilium without Envoy, AKS+Cilium, EKS Anywhere,
on-prem variants, and notably **GKE Dataplane V2 in its current
managed form** — see the docs path below) saw nothing and assumed
KubeBolt had no reliability story for them.

1.11 splits the probe in two:

- The L4 panels (Network Drops via `pod_flow_events_total{verdict="dropped"}`)
  still show whenever the L4 Hubble feed is present — the standard
  drop signal that NetworkPolicy violations + connection refused
  produce.
- The L7 panels (cluster error rate, traffic, latency, error
  hotspots) render a dedicated placeholder when L7 samples aren't
  present, with copy that names the cause (Cilium L7 proxy not
  enabled) and the affected platforms generically (EKS, AKS, on-prem,
  GKE managed) instead of accusing one platform.

The empty state is now informative instead of misleading.

### 5. Hubble GKE Dataplane V2 docs (Item 2 — docs path)

GKE's managed Dataplane V2 ships the Hubble peer with its own CA
certificate, and customers can't change that out of band. The agent
needs that CA mirrored into kubebolt-system to connect.

`deploy/helm/kubebolt-agent/README.md` now documents **three mirror
paths** ordered by operational cost:

1. **Manual one-shot copy** — single `kubectl get secret -n
   kube-system | kubectl apply -n kubebolt-system` pipe. Right for
   evaluations; fragile across cert rotation.
2. **External Secrets Operator** — declarative, survives rotation,
   single source of truth at the platform team's existing ESO
   surface.
3. **Reflector** — annotation-driven, lowest configuration. Right
   when the platform team already runs Reflector for ingress certs
   etc.

Each path includes a complete manifest, a verification step, and a
"what breaks if you skip this" callout. The init-container path
(Item 2 Path A) is deferred to the backlog — captured but not
release-scheduled until support load justifies the additional code
surface.

- **kube-prometheus-stack labelSelector mismatch flag**
  ([PR #44](https://github.com/clm-cloud-solutions/kubebolt/pull/44))
  — the agent chart's NOTES.txt now flags the case where the
  operator's existing kube-prometheus-stack release uses a
  different `app.kubernetes.io/name` label on its bundled
  node-exporter than the standalone chart does. helm dry-run
  surfaces the right copy in both branches; chart version unchanged.
- **Diagnostic logging retained on protocol boundaries.** The probes
  added during the cluster-validation campaign stay in place as
  regression sentinels (per the diagnostic-logging memory pattern).
- **Doc sync sweep.** SPEC.md, README, CLAUDE.md, deployment-scenarios,
  distribution-spec, and the chart README all updated to reflect the
  new resource-type and insights-rule counts. NetworkPolicy promoted
  out of the "describe-only" tier in SPEC.md §3.2.1 with an explicit
  cross-reference to its new first-class home in §3.2 Traffic.

## Upgrade notes

**From 1.10.x:** drop-in. No schema migration, no breaking config.
NetworkPolicy informer + insights rules light up automatically once
the chart's updated RBAC grants `list/watch networkpolicies` to the
SA. If you bind your own ClusterRole (`rbac.create=false` in values),
add this rule to keep parity:

```yaml
- apiGroups: ["networking.k8s.io"]
  resources: ["networkpolicies"]
  verbs: ["get", "list", "watch"]
```

If you also want to create / edit / delete NetworkPolicies from the
KubeBolt UI (the New-resource modal, YAML editor, Delete action),
add the write verbs too:

```yaml
- apiGroups: ["networking.k8s.io"]
  resources: ["networkpolicies"]
  verbs: ["create", "update", "delete"]
```

**Cluster-scoped tokens from 1.11 are forward-compatible.** Tokens
issued under 1.10 continue to work; they simply lack a `cluster_id`
binding and behave as "any cluster" tokens. Re-issue if you want
the per-cluster scoping.

**From pre-1.10 releases:** read the 1.10 release notes first —
this release builds on top of the 1.10 universal-data-plane
schema changes.

## Looking ahead

**1.12 is in planning.** The flagship is **Phase 6 / Mode C — Agent
reads customer's Prometheus**, closing the AMP / Azure Managed
Prometheus gap (the customer can't `remote_write` outbound from a
managed service they don't control, so KubeBolt's agent reads from
their Prom toward our VictoriaMetrics). Alongside: Pod actions
Phase 3 (Debug ephemeral container + remaining audit recs) and
`/admin/ingest-activity` live observability panel. See
`internal/roadmap-1.12.md` (internal).
