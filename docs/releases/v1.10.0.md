# KubeBolt 1.10.0 — Universal data plane + dashboard enrichments

This release is the largest change to KubeBolt's data architecture
since the project began. It closes three phases of the Universal
Data Plane Plan at once:

- **Phase 1** — every metric and label that flows into KubeBolt now
  follows Prometheus convention K8s (cAdvisor + kube-state-metrics +
  node-exporter + Hubble). One schema, multiple ingest paths.
- **Phase 2** — the agent ships an opt-in `vmagent` sidecar that
  scrapes `kube-state-metrics`, `node-exporter`, and any
  `prometheus.io/scrape`-annotated pod, and ships samples into
  KubeBolt's VictoriaMetrics via a new `POST /api/v1/prom/write`
  receiver. Operators get the depth of a full Prom stack without
  running their own Prometheus.
- **Phase 2.5** — six new operator-actionable enrichments across
  the dashboard, surfacing data the cluster already produced but
  KubeBolt didn't yet display: pod restart history, OOMKill timeline,
  node load + PSI, per-mountpoint filesystem, service endpoint
  health (UI + Insight rule), namespace ResourceQuota gauge.

Plus a customer-facing **Deployment Scenarios** doc that maps the
three most common starting states (Prom+Grafana, Prom only,
greenfield) to install recipes and feature availability per
scenario. Operators decide what to install based on what they want
to see; the doc tells them what they get.

## ⚠️ Release candidate notice

This is `1.10.0-rc.3` — a **release candidate** intended for
empirical validation across managed K8s providers (GKE, EKS, AKS) and
CNI flavors (Cilium with/without Hubble, Calico OSS, Azure CNI,
VPC CNI). The artifacts published under this tag are the bits we
intend to ship as `1.10.0` GA unless validation surfaces blockers.

RC3 supersedes RC2 — same Phase 3 + cluster-validation scope as
RC2, plus one operator-visible fix surfaced during RC2 retest of
the single-cluster self-hosted topology (BUG-2: backend installed
alongside an agent in the same cluster surfaced that cluster
twice in the UI selector).

**For prospects / evaluators:** install RC3 — it's the same code
that will be GA unless this cycle surfaces new blockers.

**For production self-hosted users:** stay on `1.9.x` until GA
cuts. Subscribe to the
[releases page](https://github.com/clm-cloud-solutions/kubebolt/releases)

### What's new in RC3 (rc.2 → rc.3)

- **Bug fix — single-cluster self-hosted duplicate cluster row
  (BUG-2).** `fix(api): skip agent-proxy auto-register when
  cluster_id matches backend's own`. When the backend runs
  in-cluster and an agent connects from the SAME cluster, the
  backend now short-circuits the agent-proxy registration —
  the cluster is already exposed via the in-cluster kubeconfig
  context, so a second registration as `agent:<UUID>` would
  surface it twice in `/api/v1/clusters` and the UI selector.
  In-vivo validated on `kind-kubebolt-dev` with the bundled
  helm install. Out-of-cluster dev runs (`make dev-clean`) are
  unaffected — the self-skip gate is empty by default. This bug
  was masked in RC1 (all agents collapsed to `cluster_id="local"`)
  and surfaced post-RC2 once the cluster_hint fix made the
  duplication visually obvious.

Everything else from RC2 still applies — see "What's new in RC2"
below for Phase 3 + the four cluster-validation fixes.

### What's new in RC2 (rc.1 → rc.2)

- **Phase 3 — customer-facing Prom `remote_write` receiver.**
  Per-tenant bearer auth (three modes: disabled / permissive /
  enforced), per-tenant rate limit (token bucket), per-tenant
  cardinality cap (VM-authoritative), admin UI for per-tenant
  overrides, `/metrics` observability surface. Operator guide:
  [`docs/integrations/prometheus.md`](../integrations/prometheus.md).
- **Bug fix — multi-cluster registry collapse (CRITICAL).**
  `fix(api)!: honor Hello.cluster_hint in resolveAgentID`. Backend
  no longer collapses N clusters to `cluster_id="local"` regardless
  of what the agent reports. Affected both OSS multi-cluster and
  SaaS ingest-token paths.
- **Bug fix — log spam in permissive auth mode.** WARN-once for
  prom remote_write fallback events; ongoing rate observable via
  `kubebolt_prom_write_requests_total{tenant_id="anonymous"}` on
  `/metrics`. One in-vivo run saw 12,880 identical WARNs from a
  single client; now 1 WARN + DEBUG-level subsequent.
- **Bug fix — helm upgrade from pre-Phase-3 release.** Nil-guard
  `.Values.tenant.id` in the agent chart templates so
  `helm upgrade --reuse-values` from `1.0.0-rc.1` doesn't panic
  with `nil pointer evaluating interface {}.id`.
- **Bug fix — node Monitor Filesystem panel empty in OSS-minimal
  installs.** P25-04 silently traded the agent's `node_fs_*` for
  node-exporter's `node_filesystem_*`, leaving installs without
  the scrape sidecar with an empty panel. PromQL `or` fallback
  restores the agent's coarse metric when node-exporter isn't
  emitting per-mountpoint detail.
- **Frontend version auto-derive.** Eliminates the manual sync
  point that caused RC1 to ship with the UI still rendering
  "1.9.0" — `apps/web/src/version.ts` now reads from
  `package.json` at build time.

Full detail and rationale in the sections below.

## Post-RC additions — Phase 3 + cluster-validation fixes

The cluster-validation campaign on top of RC1 produced two
deliverables that land before GA:

### Phase 3 — customer-facing Prom `remote_write` receiver

The Phase 2 receiver shipped with RC1 was gated only by an env
var. Phase 3 hardens it for production / SaaS:

- **Per-tenant bearer auth** with three modes — `disabled` (OSS
  default, accepts anonymous traffic), `permissive` (validates
  when present, accepts when absent under a synthetic identity),
  `enforced` (requires bearer + matching `tenant_id` label,
  rejects spoofed traffic). Switch via
  `KUBEBOLT_PROM_WRITE_AUTH_MODE`.
- **Per-tenant rate limiting** (token bucket, default 10k
  samples/s + 100k burst). When a tenant trips, the response is
  `429 Too Many Requests` with `Retry-After` honored by stock
  Prom / vmagent clients.
- **Per-tenant cardinality cap** (default 1M active series),
  enforced via a 30s periodic `count by (tenant_id)` query against
  VM. Boot-permissive: cold cache allows traffic until the first
  refresh succeeds, so backend restarts don't 413-storm the fleet.
- **Per-tenant overrides** — operators can lift or tighten the
  three knobs per tenant via the new admin UI at
  `/admin/ingest-limits`. Overrides persist in BoltDB and survive
  restarts. Save dirties only changed fields so unchanged values
  inherit fleet defaults automatically.
- **`/metrics` observability** — every request flows through a
  Prom-style counter labeled by `tenant_id` and outcome status
  (`accepted`, `rate_limit`, `cardinality`, `auth`, `body_size`,
  `malformed`, `tenant_id_mismatch`, `tenant_id_missing`,
  `injection_failed`, `upstream_error`). Active series mirrored
  via gauge. Drives the dashboards that would otherwise live in
  the log.

Full operator guide: **[`docs/integrations/prometheus.md`](../integrations/prometheus.md)**
(prerequisites, copy-paste Prom config, troubleshooting table for
every HTTP code the receiver can return).

### Bug fixes empirically surfaced by RC1 validation

The cluster-validation campaign (session `00-humo-test`) caught
four bugs that block multi-cluster deployments in their original
form:

- **`fix(api)!: honor Hello.cluster_hint in resolveAgentID`** —
  the backend collapsed every agent without an auth-set
  `ClusterID` to `cluster_id="local"` regardless of what the
  agent reported. Affected BOTH the OSS multi-cluster topology
  (multiple clusters → one backend, no auth) AND the SaaS
  `ingest-token` path (bearer doesn't populate `ClusterID`).
  Backend now honors `Hello.cluster_hint` with precedence
  `id.ClusterID > clusterHint > "local"` in both auth-disabled
  and auth-enabled branches. **Operators upgrading from RC1 with
  pre-existing `local/`-keyed `AgentRecords` in BoltDB should
  run the one-shot cleanup described in the commit message** —
  the auto-pruner only handles records with `DisconnectedAt != 0`.
- **`fix(api): WARN-once for prom remote_write permissive-fallback`**
  — the three per-request WARN logs in `permissive` mode (missing,
  empty, bad bearer) generated thousands of identical lines per
  hour from a single unauthenticated client. Replaced with one
  WARN per process pointing at
  `kubebolt_prom_write_requests_total{tenant_id="anonymous"}` for
  ongoing rate visibility; subsequent fallbacks log at DEBUG.
- **`fix(helm): nil-guard .Values.tenant in agent chart templates`**
  — `helm upgrade --reuse-values` from a pre-Phase-3 release
  (`kubebolt-agent` 0.x — no `tenant` section in user values)
  template-panicked with `nil pointer evaluating interface {}.id`
  on the new `KUBEBOLT_TENANT_ID` and `external_labels.tenant_id`
  emission sites. Without this guard, every operator upgrade
  blocks at the helm CLI before reaching the cluster.
- **`fix(web): node Monitor filesystem panel falls back to agent
  metrics when node-exporter absent`** — P25-04 silently traded
  the agent's `node_fs_used_bytes` for node-exporter's
  per-mountpoint `node_filesystem_*`, leaving OSS-minimal installs
  (agent only, no scrape sidecar) with an empty Filesystem panel.
  PromQL `or` fallback with `unless on(node)` guard restores the
  OSS-default experience without sacrificing the Prom-mature
  operator's per-mountpoint view.

## ⚠️ Breaking compatibility

**This release requires `kubebolt-agent` >= 1.0.0-rc.1.** Older
agents (0.2.x and earlier) emit the legacy schema; dashboards render
empty for any cluster they ship from. Symmetrically, running KubeBolt
1.9.x or earlier with the new agent v1.0 produces the same
empty-dashboards symptom — neither side crashes, samples still flow
into VictoriaMetrics, but the queries don't match.

Full compatibility matrix and migration steps live in
[`docs/COMPATIBILITY.md`](../COMPATIBILITY.md). Read that first if
you're upgrading an existing install.

The backend logs `WARN msg="agent below minimum version — legacy
schema"` on registration when a legacy agent connects, with a `hint`
field telling the operator exactly what to do.

## Migration

```bash
# 1. Bump KubeBolt
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
    --version 1.10.0-rc.1 \
    -n kubebolt --reuse-values

# 2. Bump every agent that ships to this backend
helm upgrade kubebolt-agent oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt-agent \
    --version 1.0.0-rc.1 \
    -n kubebolt-agent --reuse-values
kubectl rollout status ds/kubebolt-agent -n kubebolt-agent

# 3. (Optional) Enable the Phase 2 scrape sidecar — only if you
#    want KSM / node-exporter / annotated-pod metrics ingested
#    through KubeBolt's vmagent.
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
    --version 1.10.0-rc.1 \
    -n kubebolt --reuse-values \
    --set metrics.remoteWrite.enabled=true \
    --set metrics.remoteWrite.authMode=disabled
    # Production: switch authMode to permissive/enforced + provision
    # an ingest token via the integration's "Generate token" flow.

helm upgrade kubebolt-agent oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt-agent \
    --version 1.0.0-rc.1 \
    -n kubebolt-agent --reuse-values \
    --set scrape.enabled=true \
    --set scrape.remoteWriteUrl=http://kubebolt.kubebolt.svc.cluster.local/api/v1/prom/write
    # See docs/agent-scraping.md for the full config reference + the
    # kube-prometheus-stack-specific recipe (ServiceMonitor-driven
    # installs need the dedicated KSM/node-exporter scrape jobs
    # enabled explicitly).

# 4. Confirm the WARN is gone in the backend log
kubectl logs -n kubebolt deployment/kubebolt --tail=50 | grep -i "below minimum"
# (no output = both sides aligned)
```

For multi-cluster fleets, repeat steps 2-3 in every cluster. The
backend keeps working without the agent upgrade — the WARN is
informational, not a gate.

## Highlights

### Universal data plane — your existing stack, your KubeBolt

Phase 2 closes the longest-standing gap in KubeBolt's pitch: "great
UI for what the bundled agent emits — but what about everything else
my cluster already produces?" The new vmagent sidecar in the
kubebolt-agent DaemonSet scrapes:

- **kube-state-metrics** — every `kube_*` series (pod restart counts,
  endpoint readiness, ResourceQuota usage, HPA state, deployment
  rollout status, and ~140 more)
- **node-exporter** — every `node_*` series (load average, PSI
  pressure, per-mountpoint filesystem, disk IO, network detail)
- **Annotation-driven pods** — any pod carrying
  `prometheus.io/scrape: "true"` gets discovered and scraped. App
  exporters, custom metrics endpoints, all of it.

Samples ship via `POST /api/v1/prom/write` to KubeBolt's bundled
VictoriaMetrics with full Prometheus remote_write protocol support
including three-tier bearer auth (`disabled` / `permissive` /
`enforced`) — same enforcement model as the agent gRPC channel, so
operators manage one token for two ingest paths.

ADR-003 in the data plane plan: the sidecar runs **per-node** with a
local-node target filter, so an N-node cluster has N vmagents each
scraping ~1/N of the annotated targets. Same coverage, no
double-scraping, distributes load linearly with cluster size.

### Six new dashboard enrichments — Phase 2.5

Each enrichment surfaces data that existed in the cluster but was
hidden behind kubectl describes. All gated on data availability:
quiet clusters show no chip, no chart, no noise. Operator-meaningful
thresholds across the board (e.g., 80% / 95% for filesystem; 10% /
30% for PSI).

#### Pod restart history sparkline + recency badge (P25-01)

The "restarts: 47" column was lifetime-since-pod-creation and didn't
tell you whether those 47 happened today or over the last year.

- Pod list view: gated `RotateCw` icon when ≥2 restarts in 1h
  (urgent) or ≥3 in 24h (sustained), with hover tooltip showing the
  bucket breakdown (Last 1h / Last 24h / Lifetime)
- Pod detail Overview: full 24-bucket bar-chart sparkline next to
  the restart count, makes "clustered" vs "spread out" obvious

#### OOMKill timeline (P25-02)

- Pod detail Overview: gains "Last Termination" field with color-tiered
  badge (red for OOMKilled, amber for Error, neutral for Completed)
  + exit code + time-ago + responsible container name
- CapacityPage: new "Recent OOMKills" panel listing currently-OOM-
  recovered containers sorted by most recent first. Empty-state
  reads "No OOMKills detected" instead of hiding the panel — operators
  want to see silence as much as noise.

#### Node load average + PSI charts (P25-03)

NodesPage list view gains a `Load 7.62 · 7.71 · 7.85` row (1m/5m/15m)
under the CPU/Mem/Pods bars, plus a gated `Activity` icon when
PSI cpu/io/memory rate > 10% (warn) or > 30% (crit). Tooltip shows
the per-axis breakdown.

Node detail Monitor tab grows two charts: **Load average** (three
line series, 1m/5m/15m) and **PSI pressure (waiting)** with reference
lines at 10% / 30%.

#### Per-mountpoint filesystem chart (P25-04)

Node detail Monitor tab "Filesystem usage" replaces the previous
single-aggregate panel with one line per real disk mountpoint
(filtered by fstype + mountpoint regex to drop tmpfs / overlay /
kubelet per-pod volumes). Reference lines at 80% (watch) and 95%
(page). Multi-disk nodes finally show each filesystem separately
instead of summing everything into one number.

#### Service endpoint health (P25-05) — UI + Insight rule

Services list view gains an "Endpoints" column rendering
`ready / total` with severity color. ExternalName and headless
services correctly render `—` (no endpoints by design). Cluster
services with broken selectors render **red `0`** with a tooltip
explaining "Selector matches no pods — check spec.selector."

Insight rule **`service-no-endpoints`** fires (category: networking)
when a non-headless, non-ExternalName Service with a selector has
zero ready endpoints. Two failure shapes detected:
- `total = 0` → "Service has no ready endpoints" (config bug)
- `total > 0, ready = 0` → "Service has N endpoint(s) but none are
  ready" (workload bug)

#### Namespace ResourceQuota gauge (P25-06)

Dashboard NamespaceTiles: tiered chip `Q 96%` (red / amber, gated
at 70% / 85%) when the namespace's most-constrained quota crosses
the threshold. Hover surfaces full per-resource breakdown.

NamespacesPage cards grow a "Quota · <name>" strip with one mini-bar
per tracked resource (`limits.cpu`, `limits.memory`, `pods`, etc.)
showing used/hard/%. Unconstrained namespaces render an explicit
"Quota · unconstrained — No ResourceQuota bound to this namespace"
slot so cards in the same grid all share one anatomy.

### Prom-canonical schema (Phase 1 — recap)

Every UI query — every Resource Detail Monitor tab, every Cluster
Map flow rendering, every Reliability panel, the Right-Sizing P95
heuristics, the Network Drops table, the External Endpoints region
of the cluster map — now consults canonical Prometheus convention
names. Future Prom remote_write IN (Phase 3) and OTLP receivers
(Phase 4) will normalize to the same names, so a single dashboard
set covers greenfield agent installs *and* clusters that already
have Prom or OpenTelemetry shipping their own metrics.

Min-agent-version check on the gRPC channel logs a structured WARN
when a 0.2.x agent connects against a 1.10 backend, with a `hint`
field pointing operators at the upgrade command.

### Cluster map: every flow renders, including external + drops

The Phase 1 schema rename surfaced a class of UI consumers that had
been silently missing edges — five Reliability panels and three Map
data sources all joined Hubble flow data on `src_*`/`dst_*` labels.
With those fixed:
- **Traffic mode of the cluster map** correctly renders pod-to-pod
  edges
- **External endpoint nodes** render with FQDN labels when Cilium
  L7 DNS visibility is enabled (`api.github.com`, `httpbin.org`,
  etc. show up as labeled nodes in the synthetic `(external)`
  region instead of raw IPs)
- **Network Drops panel** populates with `verdict="dropped"` events,
  surfacing the early-warning channel for NetworkPolicy violations
  that the L7 panels alone miss

### Agent self-observability

The agent emits a small set of `kubebolt_agent_*` series so it shows
up in the same dashboards as the rest of the cluster:

- `kubebolt_agent_samples_collected_total` — counter
- `kubebolt_agent_samples_dropped_total` — counter
- `kubebolt_agent_buffer_size_current` — gauge
- `kubebolt_agent_buffer_size_max` — gauge
- `kubebolt_agent_memory_bytes` — gauge
- `kubebolt_agent_goroutines` — gauge
- `kubebolt_agent_info` — identity gauge=1 carrying the
  `agent_version` label so dashboards can group/filter by release

## What didn't change

- **Insights Engine** core flow: still reads `state.PodMetrics` (the
  in-memory Metrics Server cache) for its compute. Doesn't touch
  VictoriaMetrics, so the schema rename has no effect on its
  evaluations. **One new rule added:** `service-no-endpoints` in
  the new `networking` category. Engine count: 12 → 13 rules.
- Cluster connect / reconnect path: untouched.
- Auth, multi-cluster, agent-proxy SPDY tunneling, AI Copilot,
  Insights engine — all shipped in prior releases: unchanged.
- Existing dashboards (Capacity, Reliability sub-tab, Overview):
  same panels, same charts, just point at canonical schema names
  internally.

## Known limitations

- **kube-prometheus-stack KSM scraped via the dedicated agent job
  has N×duplication potential** in multi-node clusters. The bundled
  KubeBolt VictoriaMetrics ships `--dedup.minScrapeInterval=30s` by
  default to collapse duplicates server-side. External TSDB users
  must enable the same flag on their own VM/vmselect; see the
  chart README's Metrics Storage section.
- **P25-06b** (Quota usage > 85% Insight rule) is deferred to a
  future release because it requires a ResourceQuota informer
  plumbing pass that pairs with seven additional Insight rules
  planned for the next release line.
- **Calico OSS** clusters work for all features that don't need
  Cilium+Hubble; L7 telemetry (Reliability sub-tab) is unavailable
  by design. Calico-specific observability (Felix health,
  NetworkPolicy coverage analytics) is on the Phase 2.7 backlog
  pending validation campaign data across managed providers.

## What's next

Post-RC validation campaign covers GKE Dataplane V2, EKS default,
AKS default, and GKE+Calico OSS — empirical data on what works per
provider/CNI combo. Findings feed two follow-ups:

- **Phase 2.6 — Insights Engine expansion** (7 new rules including
  the deferred Quota >85%, plus rollout-stuck, PVC fill rate, PSI
  thresholds, HPA maxed-out, PDB violated)
- **Phase 2.7 — CNI expansion** (Calico Felix health + NetworkPolicy
  coverage analytics, Cilium-without-Hubble enablement guide,
  graceful degradation for CNIs without L7 telemetry)
- **Phase 4 — OTLP receiver (customer-facing)** completes the
  Universal Data Plane Plan's ingest matrix. Same per-tenant auth
  + limits as Phase 3, but speaking OpenTelemetry's wire format so
  operators with an OTel Collector pipeline can point it at
  KubeBolt without converting to Prom remote_write first.

Phase 3 itself moved out of "what's next" into this release —
see the "Post-RC additions" section above for the customer-facing
receiver + per-tenant limits UI + observability metrics.
