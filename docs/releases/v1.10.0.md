# KubeBolt 1.10.0 — Prom-canonical schema (Phase 1)

This release closes Phase 1 of the Universal Data Plane Plan — every
metric and label that flows from the agent into KubeBolt dashboards
now follows **Prometheus convention K8s** (the de-facto schema of
cAdvisor + kube-state-metrics + node-exporter + Hubble). One schema,
multiple ingest paths: future Prom remote_write and OTLP receivers
will normalize to the same names, so a single dashboard set covers
greenfield agent installs *and* clusters that already have Prom or
OpenTelemetry shipping their own metrics.

Strategic context: a self-hosted KubeBolt should integrate with the
observability stack a customer already has. Aligning to Prom
convention now lets us add Phase 2 (vmagent sidecar scraping
kube-state-metrics + node-exporter), Phase 3 (remote_write receiver),
and Phase 4 (OTLP receiver) without re-doing the dashboards.

---

## ⚠️  Breaking compatibility

**This release requires `kubebolt-agent` >= 1.0.0.** Older agents
(0.2.x and earlier) emit the legacy schema and the dashboards will
render empty for any cluster they're shipping from. Symmetrically,
running KubeBolt 1.9.x or earlier with the new agent v1.0.0 produces
the same empty-dashboards symptom — neither side crashes, samples
still flow into VictoriaMetrics, but the queries don't match.

Full compatibility matrix and migration steps live in
[`docs/COMPATIBILITY.md`](../COMPATIBILITY.md). Read that first if
you're upgrading an existing install.

The backend logs `WARN msg="agent below minimum version — legacy
schema"` on registration when a legacy agent connects, with a `hint`
field telling the operator exactly what to do.

## Migration

```bash
# 1. Bump KubeBolt
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
    --version 1.10.0 \
    -n kubebolt --reuse-values

# 2. Bump every agent that ships to this backend
helm upgrade kubebolt-agent oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt-agent \
    --version 1.0.0 \
    -n kubebolt-agent --reuse-values
kubectl rollout status ds/kubebolt-agent -n kubebolt-agent

# 3. Confirm the WARN is gone in the backend log
kubectl logs -n kubebolt deployment/kubebolt --tail=50 | grep -i "below minimum"
# (no output = both sides aligned)
```

For multi-cluster fleets, repeat step 2 in every cluster. The backend
keeps working in the meantime — the WARN is informational, not a
gate.

## What changed

### A single canonical schema across every dashboard

Every UI query — every Resource Detail Monitor tab, every Cluster
Map flow rendering, every Reliability panel, the Right-Sizing P95
heuristics, the Network Drops table, the External Endpoints region
of the cluster map — now consults the canonical names. Before this
release, those queries used names that the agent (and in some places
the backend) was the only thing emitting; future Prom/OTLP sources
would have needed translation layers to feed the same dashboards.

The **backend's PromQL label-injector** (`scopeQueryByCluster`,
`apps/api/internal/api/metrics_query.go`) is unchanged in behavior —
its bare-metric regex (`(node|pod|container|kubebolt|hubble)_*`)
already covered every canonical name. The comments and tests were
updated to reflect the new label set; the by/without-clause
regression test now uses `pod_uid` (the only `pod_*`-prefixed label
that survives) to keep the same coverage.

### Min-agent-version check at the gRPC channel

When an agent registers via the bidi gRPC channel
(`apps/api/internal/agent/server.go`), the backend compares
`Hello.AgentVersion` against `MinAgentVersion = "1.0.0"` using
semver. Below the floor, a structured `WARN` log fires with an
explicit `hint` field. Empty / unparseable versions stay silent
(test mocks, third-party forks). The check is fail-soft — the
agent still connects, samples still ingest. The check exists so the
operator notices before they spend an afternoon debugging an empty
dashboard.

### Cluster map: now renders every flow, including external + drops

A late-stage in-vivo smoke test of the schema rename surfaced a
class of UI consumers we'd missed in the initial sweep — every
panel that joined Hubble flow data on `src_*`/`dst_*` labels (the
backend's `/api/v1/flows/edges` handler, `promql.ts`'s
`collapsePodToWorkload`, and five Reliability-tab dashboards). With
those fixed:

- **Traffic mode of the cluster map** correctly renders edges between
  pod-to-pod sources and destinations. (Before the fix, the demo
  workload's `demo-load → demo-web` edges were invisible even with
  ~50 events/s reaching VM.)
- **External endpoint nodes** render with FQDN labels when Cilium
  L7 DNS visibility is enabled — `api.github.com`, `httpbin.org`
  etc. show up as labeled nodes in the synthetic `(external)`
  region instead of raw IPs.
- **Network Drops panel** (Reliability tab) populates with row pairs
  derived from `verdict="dropped"` events, surfacing the early-warning
  channel for NetworkPolicy violations that the L7 panels alone miss.

### Agent self-observability

The agent emits a small set of `kubebolt_agent_*` series via a new
self-collector goroutine, so it shows up in the same dashboards as
the rest of the cluster:

- `kubebolt_agent_samples_collected_total` — counter
- `kubebolt_agent_samples_dropped_total` — counter
- `kubebolt_agent_buffer_size_current` — gauge
- `kubebolt_agent_buffer_size_max` — gauge
- `kubebolt_agent_memory_bytes` — gauge (`runtime.MemStats.Alloc`)
- `kubebolt_agent_goroutines` — gauge (`runtime.NumGoroutine()`)
- `kubebolt_agent_info` — identity gauge=1 carrying the
  `agent_version` label so dashboards can group/filter by release

## What didn't change

- Insights Engine: still reads `state.PodMetrics` (the in-memory
  Metrics Server cache) for its 12 rules. Doesn't touch
  VictoriaMetrics, so the schema rename has no effect on its
  evaluations.
- Cluster connect / reconnect path: untouched.
- Auth, multi-cluster, agent-proxy SPDY tunneling, all shipped in
  prior releases: unchanged.
