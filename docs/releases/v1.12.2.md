# KubeBolt 1.12.2 — fix the update-check stamp on container images

Single-fix patch. The container image at
`ghcr.io/clm-cloud-solutions/kubebolt/api:1.12.1` (and every earlier
self-hosted image since the update-check feature shipped in 1.12.1)
was compiled **without** the `-ldflags="-X main.version=<tag>"`
stamp the standalone binaries get, so `main.version` (in
`apps/api/cmd/server/main.go`) kept its default `"dev"`. The
update-check package short-circuits on a `"dev"` version, which
means **every self-hosted install since 1.12.1 has surfaced
`{"currentVersion":"dev","isUpdateAvailable":false}` from
`/api/v1/update-check` and never rendered the Topbar update chip**.

The 1.12.1 release was specifically designed as the carrier vehicle
for the 1.13 announcement chip. The chip works fine in dev (where
the version stamp comes from `go build -ldflags`) and on
homebrew/krew binaries (the release workflow stamps them at line 501
of `release.yml`) — but on the image path it was silently broken.
1.12.2 closes that gap so the 1.13.0 chip actually reaches users on
upgrade.

---

## What ships

### Backend

- **`apps/api/Dockerfile`** — adds `ARG VERSION=dev` and rewires the
  `go build` to `-ldflags="-s -w -X main.version=${VERSION}"`. The
  default `dev` keeps `docker build` from a working tree behaving as
  before; the release workflow overrides it with the tag.
- **`.github/workflows/release.yml` `build-api` job** — passes
  `build-args: VERSION=${{ steps.meta.outputs.version }}` to
  `docker/build-push-action`. Resolved value comes from
  `docker/metadata-action`'s `version` output, which strips the `v`
  prefix from the tag (`v1.12.2` → `1.12.2`).

The single-container image (`Dockerfile.single`) is unaffected — it
already consumed a pre-built binary from `build-binaries`, which
already had the `-ldflags` stamp.

The web image and Helm chart are unchanged on the binary side; the
chart's `appVersion` bumps from `1.12.1` to `1.12.2` purely to
match the image tag.

---

## Verification

After upgrading the backend image to `1.12.2`:

```bash
curl -sH "Authorization: Bearer <admin-jwt>" \
  https://<your-host>/api/v1/update-check | jq
# Expected:
# {
#   "enabled": true,
#   "currentVersion": "1.12.2",
#   "latestVersion": "1.13.0",
#   "isUpdateAvailable": true,
#   "releaseUrl": "https://github.com/clm-cloud-solutions/kubebolt/releases/tag/v1.13.0",
#   "releaseName": "v1.13.0",
#   "publishedAt": "2026-05-28T..."
# }
```

The Topbar update-check chip should render `Update v1.13.0
available` within the next 6-hour cache cycle (or immediately on a
fresh pod start).

## Upgrade

```bash
helm upgrade kubebolt oci://ghcr.io/clm-cloud-solutions/kubebolt/helm/kubebolt \
  --version 1.12.2 \
  --reuse-values
```

Zero-config impact — same chart values, same env vars, same RBAC.
The only behavioral change is the surfaced version string.

---

## What does NOT ship in 1.12.2

- **Mode C / promread** — that's 1.13.0+. This patch keeps you on
  the 1.12 line; upgrade to 1.13.0 separately when you're ready.
- **Hubble default flip** — 1.13.0+ flips
  `kubebolt-agent` `hubble.enabled` from `true` to `false` and is
  marked BREAKING in the 1.13.0 release notes. 1.12.2 keeps the
  1.12.x default of `true`.

---

## Forward-port to 1.13.x

The same Dockerfile + workflow fix needs to land on `main` (which is
now at `1.13.0`). 1.13.0's image was cut with the same broken
release.yml and will report `"currentVersion":"dev"` from
`/api/v1/update-check` until 1.13.1 ships. Tracked separately — the
1.13.1 patch will forward-port these two changes and cut a fresh
v1.13.0-equivalent image with the stamp.
