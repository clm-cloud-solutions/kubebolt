After ~1 scrape interval ({{ default "30s" .Values.scrape.scrapeInterval }}), verify targets
are UP via the vmagent admin UI:

  POD=$(kubectl -n {{ .Release.Namespace }} get pod -l app.kubernetes.io/name={{ include "kubebolt-agent.name" . }} -o jsonpath='{.items[0].metadata.name}')
  kubectl -n {{ .Release.Namespace }} port-forward $POD 8429:8429 -c vmagent &
  curl -s http://localhost:8429/api/v1/targets | jq '.data.activeTargets[] | {job,health}'

If a job you expected to see (node-exporter, kube-state-metrics) is
missing from activeTargets, the most common cause is a label-selector
mismatch. The defaults in values.yaml match standalone Prometheus
charts; kube-prometheus-stack uses
`app.kubernetes.io/name=prometheus-node-exporter` for node-exporter
(NOT the default `node-exporter`). Inspect the comment block on
scrape.discovery in values.yaml for the override pattern.
{{- end }}

⚠️  IF your cluster does NOT run Cilium, the agent will dial the
Hubble relay forever and emit one WARN per node every ~1 minute
("name resolver error: produced zero addresses"). Verify Cilium
is actually present before leaving hubble.enabled=true:

  kubectl -n kube-system get pods -l k8s-app=cilium
{{- else }}

To enable Hubble flow collection (requires Cilium with Hubble relay
reachable at hubble-relay.kube-system.svc.cluster.local:80 — or a
custom hubble.relay.address override):

  helm upgrade --reuse-values --set hubble.enabled=true ...

Quick check whether Cilium is present in your cluster:

  kubectl -n kube-system get pods -l k8s-app=cilium

Without Cilium the Reliability dashboard's L7 HTTP + Network Drops
panels auto-hide; no feature loss on non-Cilium clusters.
